## 本章背景知识
主要介绍kingbase RAC 系统架构。
### 概述
1、Clusterware 是什么？[^1]
是一种全局资源管理系统，主要是解决大部分分布式系统中出现的，如脑裂、集群节点的增减问题。
2、Clusterware 集群能够提供以下功能[^2]：
（1）高吞吐
（2）高压力
（3）高负载
### kingbase RAC 术语
#### Resources （资源）
组成集群所有节点上的数据库实例，VIP，内存，共享磁盘 都称之为资源，集群需要统一调度管理使用多个节点上的资源。
#### Fence agents（隔离客户端）
发送给需要被隔离设备相关信号和脚本，收到执行命令之后，此节点将被执行重启操作。[^3]
#### Corosync-qdevice （守护进程）
1、Corosync-qdevice[^4]是在集群的每个节点上运行的守护进程。
2、Corosync-qdevice 在某个（几个）节点故障后，发起选举投票，是否踢出有问题的节点。
3、建议在节点数为偶数的集群中使用，强烈建议在节点数为2的集群中使用。
#### Corosync-qnetd（守护进程）
1、Corosync-qnetd是一个运行在集群外部的守护进程，其目的是向corosync-qdevice提供网络投票相关功能，相当于witness 。
2、Corosync-qnetd 同时支持多个集群。

#### Cluster membership layer（集群节点关联层）
该组件提供关于集群的消息传递、成员关系和仲裁信息。目前，Pacemaker支持Corosync作为该层。
####  Cluster resource manager（集群资源管理器）
1、集群资源管理其提供自动处理集群中发生的故障的功能，并管理集群资源。
2、资源包括：管理VIP 、共享磁盘、PINGD（监控）、数据库实例。
3、集群故障包括：添加节点、剔除节点、集群维护、集群硬件损坏等行为。


### kingbase Clusterware 系统架构
#### Clusterware  组件
1、Clusterware 组件主要由两大组件组成。
（1）Pacemaker 集群资源管理组件。
（2）Corosync 集群引擎组通信组件。
2、Clusterware 组件的功能[^5]。
（1）Clusterware 组件主要提供集群管理服务功能。
（2）Clusterware 组件需要部署到集群的每个节点。
（3）Clusterware 组件需要提前准备多节点共享的存储设备。

#### Pacemaker
Pacemaker是一种高可用性集群资源管理器[^8] [^9]
1、Pacemaker提供处理集群中发生的事件并对其做出反应的中枢大脑，保持完整性并尽量减少所需服务(资源)的停机时间。由ClusterLabs[^10]社区维护。
2、 Pacemaker 具有以下功能：[^11]
（1）检测节点级和服务级故障并从故障中恢复  。
（2）通过隔离（Fencing）故障节点确保数据完整性的能力  。
（3）支持集群包含一个或多个节点  。
（4）支持多种资源管理的管理接口标准(任何软件[^12]的都可以进行集群化)。
（5）支持共享存储。
（6）支持多种集群架构（主/备模式，N+1模式，RAC 模式）
（7）支持从任何节点自动配置整个集群。
（8）支持高级服务类型：支持克隆和容器化服务。
（8）提供统一的，简单的集群管理工具。

> [!note]
> 隔离（Fencing）[^13]，也被称为STONITH (Shoot The Other Node In The Head的缩写)，
> 1、是一种强制隔离节点功能。
> 2、例如：
> （1）切断目标电源的智能电源开关，
> （2）切断目标对本地网络的访问的智能网络交换机。  
> 3、Pacemaker 可以将隔离的设备作为一种特殊的资源类别。  
> 4、没有隔离措施（Fencing），集群可能无法从某些故障节点中安全恢复，造成整个集群都不可用。

3、【crmsh】配置Pacemaker 管理的资源的工具，主要是以下功能：
（1）集群的VIP卸载和加载。
（2）集群数据库存储空间资源分配。
（3）集群的启动和停止。
（4）集群隔离、添加和剔除节点。

#### Corosync

1、【Corosync】是一种组（groups)通信系统[^14]，可以实现应用程序高可用性的功能[^15]。
2、【Corosync】提供关于集群的消息传递、成员关系和仲裁信息。Pacemaker使用Corosync作为集群节点的通信组件。
3、【Corosync】最初来源于[Vision - OpenAIS](http://openais.eu/en/vision/)项目，并根据新的BSD许可证进行许可。
4、【Corosync】的更多详情请参见[Corosync官方地址](https://github.com/corosync/corosync)

#### Resource Agents
`Resource Agents`是一个操作系统组件，可以启动、停止和监视节点。Pacemaker 主要是从`Resource Agents` 获取到主机的相关信息。[^16]
#### Distributed Lock Manager, DLM（集群文件系统)
1、Distributed Lock Manager, DLM（集群文件系统)指的是集群文件系统都使用通用的分布式锁管理器(Distributed Lock Manager, DLM)。
2、DLM直接使用Corosync实现消息传递和节点功能。

#### pacemakerd
用于启动和监控后台辅助守护进程。在意外终止情况下并重新启动后台守护进程[^17]。
#### pacemaker-based 
1、Cluster Information Base(CIB)  是用XML[^18] 格式保存所有节点和资源状态信息的一个文件。
2、pacemaker-based 进程用于在整个集群保证CIB 信息同步。
3、CIB包括以下内容：
（1）节点IP地址。
（2）内存信息。
（3）节点运行是否正常。
（4）节点是否被隔离。
#### pacemaker-attrd 
1、pacemakder-attrd 用于维护所有节点的属性信息。
2、当某个节点的属性发生变化，pacemak-attrd 负责将信息传递给整个集群。保持信息的同步。
3、属性信息
（1）节点IP地址。
（2）操作系统版本。
（3）节点运行是否正常。
（4）节点是否被隔离。
#### pacemaker-schedulerd
用于调度集群什么时候该执行哪些命令。
#### pacemaker-execd
 用于集群执行操作的执行器。
 1、由`pacemaker-controld` 告诉`pacemaker-execd` 执行什么样的操作。
 2、`pacemaker-execd`的隔离节点操作是交给`pacemaker-fenced` 进程所执行。
#### pacemaker-fenced
此守护进程专门用于剔除和隔离节点。
#### pacemaker-controld
控制器进程，维护节点信息（CIB）的一致性，并协调其他组件。
#### Pacemaker HA
Pacemaker 只通过一个节点发布集群的决策和操作，如果当前节点出问题。会通过CIB信息文件快速通过另外一个节点进行发布。



#### pacemaker_remote

一个常规群集最多只能包含 32 个节点。借助 `pacemaker_remote` 服务，可以将高可用性群集进行扩展，使其包含超出此限制的额外节点。

`pacemaker_remote` 服务可作为物理节点（称为远程节点）运行，也可作为虚拟节点（称为 Guest 节点）运行。与常规群集节点不同，远程节点和 Guest 节点都作为资源由群集进行管理。因此，它们不会计入群集堆栈的 32 个节点限制。不过，从资源管理的角度而言，它们的行为与常规群集节点无异。

对远程节点不需要安装完整的群集堆栈，因为它们只运行 `pacemaker_remote` 服务。该服务充当代理，可让“常规”群集节点上的群集堆栈连接到该服务。因此，运行 `pacemaker_remote` 服务的节点可作为远程节点有效集成到群集中。
